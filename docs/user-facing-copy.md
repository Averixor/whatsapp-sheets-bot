# Вимоги до користувацьких текстів, назв аркушів та технічних даних

Політика UX-копірайту для WASB: що бачить користувач у sidebar, меню, діалогах, health-звітах і назвах аркушів. Технічні ідентифікатори в коді (`CONFIG`, `SheetSchemas_`, `DataAccess_.getSheet`) — окремо від цього документа.

Пов’язано: [`verify-no-russian-text.mjs`](../scripts/verify-no-russian-text.mjs) (заборона російських маркерів у проєктних текстах), [`verify-user-facing-copy.mjs`](../scripts/verify-user-facing-copy.mjs) (заборона технічних токенів у користувацьких рядках; `npm run ci` після `verify-no-russian-text`), [`.cursor/rules/user-facing-copy.mdc`](../.cursor/rules/user-facing-copy.mdc) (правило для агентів).

---

## 1. Загальний принцип

Усі тексти, які бачить або використовує користувач, мають бути написані **українською мовою**, бути зрозумілими та **не містити технічного навантаження**.

Користувач повинен бачити прості назви дій, повідомлень і розділів, а не внутрішню технічну реалізацію системи.

Технічні назви, службові ідентифікатори, назви функцій, модулів, змінних, перевірок, службових діапазонів та внутрішніх аркушів **не повинні відображатися** в користувацькому інтерфейсі.

---

## 2. Що вважається користувацьким текстом

До користувацьких текстів належать:

- назви кнопок;
- пункти меню;
- заголовки блоків;
- підказки;
- спливаючі повідомлення;
- повідомлення про помилки;
- повідомлення про успішне виконання дій;
- тексти в діалогових вікнах;
- назви розділів бокової панелі;
- назви користувацьких аркушів;
- будь-які інші тексти, які бачить користувач під час роботи з таблицею або панеллю.

Усі такі тексти мають бути українською мовою та сформульовані зрозуміло для звичайного користувача.

---

## 3. Заборонено в користувацькому інтерфейсі

У користувацькому інтерфейсі заборонено використовувати:

- технічні назви аркушів;
- назви функцій;
- назви модулів;
- назви змінних;
- назви внутрішніх перевірок;
- англомовні службові терміни;
- скорочення, зрозумілі лише розробнику;
- назви з технічного завдання, якщо вони не призначені для користувача;
- внутрішні ідентифікатори системи.

Приклади заборонених текстів:

- `SEND_PANEL`
- `PERSONNEL`
- `DICT_SUM`
- `VACATION_CHECK`
- `CONFIG`
- `DATA`
- `LOGS`
- `reloadData`
- `rebuildPanel`
- `checkStatus`
- `handler`
- `module`
- `validation failed`
- `ARRAYFORMULA`

---

## 4. Правильні користувацькі формулювання

Замість технічних назв мають використовуватися прості українські формулювання.

| Заборонено (приклад) | Дозволено |
| -------------------- | --------- |
| `Перечитати SEND_PANEL` | `Оновити дані` |
| `Пересобрати панель` (з технічним контекстом) | `Оновити панель` |
| `SEND_PANEL` | `Панель надсилання` |
| `PERSONNEL` | `Особовий склад` |
| `DICT_SUM` | `Довідник підсумків` |
| `VACATION_CHECK` | `Перевірка відпусток` |
| `CONFIG` | `Налаштування` |
| `LOGS` | `Журнал дій` |
| `validation failed` | `Є помилки в даних` |
| `send all records` | `Надіслати всі` |
| `reload sheet data` | `Оновити дані з таблиці` |

---

## 5. Назви аркушів

Назви аркушів Google Таблиці, які **бачить або використовує користувач**, мають бути написані українською мовою та не повинні містити технічних назв, внутрішніх ідентифікаторів або службових скорочень.

Користувацькі аркуші повинні мати зрозумілі назви, наприклад:

- `Особовий склад`
- `Панель надсилання`
- `Відпустки`
- `Перевірка відпусток`
- `Зведення`
- `Налаштування`
- `Журнал дій`

Технічні назви аркушів допускаються лише для **службових** технічних аркушів, які не призначені для звичайного користувача.

---

## 6. Технічні аркуші

Усі технічні дані, службові таблиці, внутрішні перевірки, допоміжні діапазони, системні налаштування, журнали, проміжні розрахунки та інші елементи технічної реалізації мають зберігатися лише на **технічних аркушах**.

Технічні аркуші мають бути відокремлені від користувацьких аркушів.

На користувацьких аркушах не повинно бути службових назв, внутрішніх кодів, технічних колонок або допоміжних розрахунків, які не потрібні користувачу для роботи.

Якщо технічні дані необхідні для роботи формул, Apps Script або автоматичних перевірок, вони мають бути винесені на окремі технічні аркуші.

Технічні аркуші бажано **приховувати** від користувача, якщо вони не потрібні для щоденної роботи.

---

## 7. Де дозволені технічні назви

Технічні назви, службові терміни та внутрішні ідентифікатори дозволено використовувати лише в таких місцях:

- у коді Apps Script;
- у технічній документації;
- у службових технічних аркушах;
- у логах;
- в автоматичних перевірках;
- у тестах;
- у внутрішніх налаштуваннях системи.

Вони **не повинні** потрапляти в інтерфейс, меню, кнопки, повідомлення, назви користувацьких аркушів або тексти, які бачить користувач.

---

## 8. Вимоги до повідомлень

Усі повідомлення системи мають бути українською мовою та зрозумілими без знання технічної частини.

**Успіх** — пояснює результат дії:

- `Дані оновлено`
- `Панель оновлено`
- `Повідомлення надіслано`
- `Усі доступні записи надіслано`

**Помилки** — людською мовою:

- `Є помилки в даних`
- `Не заповнено обов’язкові поля`
- `Неможливо надіслати повідомлення`
- `Не знайдено потрібні дані`
- `Перевірте таблицю та спробуйте ще раз`

Заборонено показувати користувачу сирі технічні помилки, назви функцій, службові коди або внутрішні ідентифікатори.

---

## 9. Критерії приймання

Реалізація вважається прийнятою, якщо:

1. Усі кнопки, меню, заголовки, підказки, повідомлення та діалогові вікна написані українською мовою.
2. У користувацькому інтерфейсі відсутні технічні назви, англомовні службові терміни, назви функцій, модулів, змінних та внутрішніх перевірок.
3. Назви користувацьких аркушів написані українською мовою та зрозумілі користувачу.
4. Усі технічні дані винесені на окремі технічні аркуші.
5. Користувацькі аркуші не містять службових колонок, внутрішніх кодів або допоміжних розрахунків, які не потрібні користувачу.
6. Технічні аркуші відокремлені від користувацьких аркушів і, за можливості, приховані.
7. Спливаючі повідомлення, повідомлення про помилки та повідомлення про успішні дії не містять технічних деталей.
8. Перед перейменуванням аркушів перевірено залежності у формулах, Apps Script-коді, тестах і документації.

---

## 10. Основна вимога

Усе, що бачить користувач, має бути **українською мовою**, **простим** і **зрозумілим**.

Усе технічне має залишатися тільки в коді, документації, автоматичних перевірках або на технічних аркушах.

Це **цільова політика**. Повна відповідність усьому коду — окремі PR, не змішувати з structural moves.

| Область | Зараз | Ціль |
| ------- | ----- | ---- |
| Фізичні назви вкладок | `PERSONNEL`, `SEND_PANEL`, `DICT_SUM`, … у `CONFIG` / `SheetSchemas_` | Технічні ключі в коді можуть лишатися; **перейменування вкладок** — окрема міграція з аудитом формул і CI |
| Sidebar / `ui/Js.*.html` | UA copy; CI guard `verify-user-facing-copy.mjs` (#37) | Тримати без технічних витоків |
| Health / diagnostics UI | UA titles/details (`diagnostics/Diagnostics.Health.gs`, …) | Без `CONFIG`/`PHONES` у текстах для оператора |
| Меню таблиці | `WASB` → `Відкрити панель` — ок | Без технічних підменю |
| Зведення дня | Підписи без `_` (див. `AGENTS.md`) | Тримати |

**Файли з користувацьким текстом (пріоритет рев’ю):** `ui/Sidebar.html`, `ui/JavaScript.html`, `ui/Js.*.html`, `ui/Styles*.html`, `core/Code.gs` (меню/alert), `reports/Summaries.gs`, `sendpanel/SendPanel.gs`, health renderers у `diagnostics/`.

**Не чіпати без окремого ADR:** глобальне перейменування `CONFIG.*_SHEET`, `SheetSchemas_` keys, посилання в формулах на вкладках `01`…`12`.
