# Техническое задание
## Этап 4.1 — интеграционное завершение и cleanup Stage 4 Google Apps Script системы sidebar, SEND_PANEL, сводок и карточек бойцов

## Формат работ

Без изменения основной бизнес-логики, без перехода на другую платформу, без миграции на внешний backend, без радикального редизайна интерфейса, с сохранением текущих пользовательских сценариев и визуального UX.

---

## 1. Цель работ

Провести этап 4.1 после внедрения базового Stage 4 для того, чтобы:

- довести Stage 4 от состояния **backend-first** до состояния **полноценно интегрированного прикладного слоя**;
- перевести client-side и sidebar на **canonical Stage 4 entrypoints**, а не на смесь старых и новых вызовов;
- устранить дубли публичных server-side функций и helper-логики;
- сократить transitional/legacy-хвосты, которые мешают сопровождению;
- формализовать единую точку входа для sidebar-сценариев;
- убрать архитектурную “двойную бухгалтерию” между старым SidebarServer API и Stage 4 API;
- подготовить проект к нормальной работе через **GitHub + clasp**, включая manifest;
- очистить и упорядочить документацию, чтобы в проекте оставались только эксплуатационно полезные документы;
- завершить stage 4 так, чтобы следующая доработка шла уже от чистой базы, а не от компромиссной полуинтеграции.

После реализации Stage 4 в проекте уже присутствуют:

- `Stage4ServerApi.gs`

- `Validation.gs`
- `AuditTrail.gs`

Также реализованы:

Однако текущая реализация всё ещё имеет важные ограничения:

- manifest (`appsscript.json`) отсутствует в архиве;

- transitional слой всё ещё шире, чем должен быть для эксплуатационно зрелой базы.

То есть проект уже стал лучше, но пока ещё местами похож на аккуратно выровненный блиндаж, в котором старые кабели всё ещё висят рядом с новыми.

## 3. Объём работ

### 3.1. В объём работ входит

- полная client-side интеграция Stage 4 API;
- перевод sidebar-вызовов на canonical Stage 4 сценарии;
- cleanup и дедупликация `SidebarServer.gs`;
- устранение дублирующихся helper-функций;
- формализация единого публичного server-side слоя для sidebar;
- добавление `appsscript.json` и подготовка проекта к `clasp`/GitHub workflow;
- инвентаризация и упрощение transitional compatibility layer;
- нормализация документации;
- расширение тестов на интеграционный слой;

- реорганизация проекта “ради красоты”, если это не улучшает поддержку и эксплуатацию.

## 4. Основные проблемы, которые должен решить этап

### 4.1. Неполная интеграция Stage 4 в UI

На client-side уже существуют:

- `Stage4Api`
- `UiState`
- `LoaderService`
- `ToastService`
- `ErrorPresenter`
- `RenderCoordinator`
- `ActionDispatcher`

Но значительная часть UI всё ещё живёт по Stage 3/legacy-паттерну.

Требуется:

- перевести ключевые пользовательские действия на `Stage4Api`;
- сделать `ActionDispatcher` и `RenderCoordinator` фактически рабочими координаторами, а не декларативной прослойкой;
- убрать хаотичные прямые вызовы старых API из UI, где уже существует stage 4 use-case.

### 4.2. Двойная бухгалтерия в `SidebarServer.gs`

В `SidebarServer.gs` сохраняются дубли и переопределения одних и тех же публичных функций.

Это создаёт риски:

- поддержки “не той” версии функции;
- путаницы в отладке;
- скрытых конфликтов между старым и новым поведением;
- накопления нового legacy.

Требуется:

- оставить **одну canonical реализацию** каждой публичной sidebar-функции;
- старые функции либо удалить, либо перевести в явный compatibility wrapper;
- исключить неявное переопределение функций в одном файле.

### 4.3. Дубли helper-функций

В проекте сохраняется дублирование утилит, в частности HTML escaping и смежных helper-методов.

Требуется:

- определить canonical helper/source-of-truth;
- удалить или закрыть вторичные дубли;
- запретить дальнейшее размножение одинаковых helper-ов.

### 4.4. Отсутствие manifest

Проект пока неполон как нормальная кодовая база для GitHub/clasp-ведения.

Требуется:

- добавить `appsscript.json`;
- описать необходимые manifest-настройки;
- обеспечить совместимость структуры проекта с выгрузкой/загрузкой через `clasp`.

### 4.5. Шум в документации

Количество документов выросло, но часть из них уже начинает мешать.

Требуется:

- оставить только документы, нужные для реальной работы;
- архивные/переходные отчёты либо вынести в отдельный блок, либо пометить как historical;
- исключить ситуацию, когда разработчик не понимает, какой документ главный.

## 5. Функциональные требования

### 5.1. Полная client-side интеграция Stage 4

Необходимо завершить подключение Stage 4 в `JavaScript.html`.

#### Требуется:

- все ключевые сценарии записи и orchestration-вызовы выполнять через `Stage4Api`;
- `ActionDispatcher` должен стать основной точкой запуска пользовательских действий;
- `RenderCoordinator` должен в явном виде определять, какие зоны UI обновлять после конкретного сценария;
- `UiState` должен использоваться как единый источник client-side состояния;
- `ToastService`, `LoaderService`, `ErrorPresenter` должны применяться последовательно, а не выборочно.

#### Минимально перевести на canonical Stage 4 flow:

- генерацию SEND_PANEL на дату;
- генерацию SEND_PANEL на диапазон;
- массовую отметку строк как отправленных;
- массовое снятие статуса отправки;
- отправку pending rows;
- сбор дневной сводки;
- сбор детальной сводки;
- открытие карточки бойца;
- проверку отпусков и дней рождения;
- создание следующего месяца;
- reconciliation и maintenance actions.

#### Требование:

UI не должен собирать сценарий из случайной цепочки вызовов.  
UI должен вызывать один верхнеуровневый сценарий и получать единый результат.

### 5.2. Cleanup публичного sidebar API

Необходимо провести зачистку `SidebarServer.gs` и связанных server-side entrypoints.

#### Требуется:

- убрать дубли публичных функций;
- для каждой публичной функции определить:
  - canonical entrypoint;
  - compatibility status;
  - слой, к которому она относится;
- убрать неявное переопределение функций;
- сделать явную схему:
  - `Code.gs` — центральный фасад/entrypoint;
  - `Stage4ServerApi.gs` — canonical public application scenarios;
  - `SidebarServer.gs` — тонкий compatibility или transport layer, если он вообще нужен;
- зафиксировать, какие функции остаются публичными для UI, а какие являются внутренними.

#### Минимально нормализовать:

- `getMonthsList`
- `getSidebarData`
- `generateSendPanelSidebar`
- `getSendPanelSidebarData`
- `getDaySummaryByDate`
- `getDetailedDaySummaryByDate`
- `checkVacationsAndNotifySidebar`
- `createNextMonthSheetSidebar`
- `switchBotToMonthSidebar`
- `markMultipleAsSentFromSidebar`

### 5.3. Дедупликация helper-слоя

Необходимо привести helper-слой к одному каноническому виду.

#### Требуется:

- инвентаризировать дубли:
  - `escapeHtml_`
  - date helpers
  - response helpers
  - misc utility helpers
- выбрать canonical implementation для каждого типа helper-а;
- заменить дубли на wrapper либо удалить их;
- дополнить deprecated registry для временно сохраняемых переходных alias-функций;
- запретить прямое размножение одинаковых утилит в разных файлах.

#### Требование:

в проекте не должно быть нескольких равноправных реализаций одного и того же helper-а.

### 5.4. Manifest и GitHub/clasp-ready структура

Необходимо подготовить проект к нормальному ведению как исходного кода.

#### Требуется:

- добавить `appsscript.json`;
- описать:
  - timezone;
  - exception logging;
  - runtime;
  - scopes;
  - webapp / execution settings, если применимо;
- обеспечить, чтобы архив/репозиторий содержал полную структуру для `clasp push/pull`;
- при необходимости добавить:
  - `.clasp.json.example`
  - `README` по локальной работе через clasp.

#### Требование:

проект должен быть пригоден для хранения в GitHub без “невидимой половины” конфигурации.

### 5.5. Нормализация документации

Необходимо сократить документационный шум и выделить главный набор документов.

#### Обязательный набор документов после cleanup:

- `ARCHITECTURE.md`
- `PUBLIC_API_STAGE4.md`
- `RUNBOOK.md`
- `CHANGELOG_STAGE4.md` или `STAGE4_REPORT.md`
- `README.md` с краткой картой проекта

#### Дополнительно:

- historical-документы могут быть оставлены, но должны быть:
  - перенесены в `/docs/archive/` либо
  - явно помечены как legacy/historical.

#### Требование:

документация должна помогать работать, а не устраивать археологические раскопки.

### 5.6. Усиление integration tests

Нужно расширить тесты именно на проблемную зону интеграции stage 4.

#### Минимально проверить:

##### UI/API integration tests
- вызов Stage4Api для генерации панели;
- вызов Stage4Api для массовой отметки строк;
- корректный разбор server contract в UI;
- правильный toast/error/loading flow;
- корректная post-operation sync логика.

##### Sidebar API consistency tests
- отсутствие дублей публичных функций;
- соответствие публичных entrypoints документации;
- отсутствие неявного переопределения.

##### Helper consistency tests
- единая canonical helper-реализация;
- deprecated wrapper-ы работают и не ломают совместимость.

##### Manifest/project structure tests
- наличие manifest;
- полнота структуры проекта для clasp-compatible поставки.

## 6. Нефункциональные требования

### 6.1. Совместимость

После выполнения этапа без деградации должны продолжать работать:

- генерация SEND_PANEL;
- отправка невідправлених;
- дневная сводка;
- детальная сводка;
- карточка бойца;
- календарь выбора даты;
- проверка отпусков;
- проверка дней рождения;
- создание следующего месяца;
- переключение активного месяца;
- диагностика;
- health check;
- reconciliation;
- batch actions;
- trigger jobs.

### 6.2. Поддерживаемость

Новые функции должны:

- добавляться через use-case и orchestration layer;
- не обходить Stage 4 API прямыми legacy-вызовами;
- не плодить новые публичные entrypoints без необходимости;
- не создавать новые helper-дубли;
- не размывать границу между UI, server API, use-cases и repositories.

### 6.3. Диагностируемость

Должно быть легко понять:

- какой сценарий вызван из UI;
- какой public API entrypoint был использован;
- какой use-case реально отработал;
- какой contract вернулся в UI;
- какая post-sync логика была выполнена;
- был ли использован compatibility wrapper;
- относится ли ошибка к UI coordination, API routing, validation или данным.

### 6.4. Чистота архитектуры

После этапа проект не должен содержать:

- неявно переопределяемых публичных функций;
- нескольких конкурирующих public API для одной и той же операции;
- дублирующих helper-функций без явного статуса deprecated;
- документов, в которых непонятно, какой из них основной.

## 9. Критерии приёмки

Работа считается принятой, если одновременно выполняются все условия:

- Stage 4 используется не только на backend, но и как основной прикладной слой для UI;
- ключевые клиентские сценарии вызываются через `Stage4Api`;
- в `SidebarServer.gs` отсутствуют опасные дубли и неявные переопределения публичных функций;
- helper-дубли устранены либо формально переведены в deprecated wrappers;
- в проекте присутствует `appsscript.json`;
- структура проекта пригодна для GitHub/clasp-ведения;
- документация сокращена до понятного рабочего набора;
- diagnostics умеет выявлять дубли, legacy-routing и structural issues;
- regression/integration tests покрывают cleanup-слой;
- текущий UX не деградировал;
- бизнес-логика существующих сценариев не изменилась;
- проект стал заметно чище, понятнее и безопаснее для дальнейшей доработки.
